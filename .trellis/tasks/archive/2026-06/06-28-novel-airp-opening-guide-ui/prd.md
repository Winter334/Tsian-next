# 小说 AIRP 开局向导 UI 范式

Parent: `06-27-default-card-novel-reader-airp`

## Goal

为 `apps/play-frontend-dev` 的小说 AIRP 开局流程定义并实现一套可扩展 UI 范式，使当前“导入小说”步骤和后续角色初始化、游玩倾向选择、开局确认等步骤能共用同一套向导容器、流程预览、导航按钮和视觉语言。

## Problems Observed

- 当前导入向导没有彻底隔离正式游戏界面，视觉上会露出后面的游戏/剧情区域，导致用户不确定自己处于“开局流程”还是“正式游玩”。
- 当前向导框像一个普通表单，层级、留白、信息组织和行动按钮都不够好看，不像 Tsian 默认小说 AIRP 的首屏体验。
- 当前 UX 缺少明确的“上一步 / 下一步 / 当前步骤”概念，也没有流程预览，后续接入多步开局流程会变得混乱。
- 当前导入成功后只显示状态变化，用户无法看到章节切分结果，也无法预览切分效果是否合理。
- 当前导入步骤把粘贴文本和选择文件堆在同一屏，像工程表单；应先让玩家选择导入方式，再进入对应方式的输入界面。

## Product Decisions From Prior Task

- 平台负责存档管理；游戏卡前端只服务当前 save，不再做新游戏/继续游戏。
- 首屏长期方向是开局动画，但本任务暂不实现动画。
- 开局流程采用“向导式 UI + architect 对话”的混合形态：结构性选择用 UI，开放性创作用 Agent 对话。
- 当前已完成的导入能力是开局流程第一步；后续会接入初始抽取、原著/原创角色选择、原创角色访谈、游玩倾向、开局确认等步骤。
- 本任务只维护 `apps/play-frontend-dev`，不修改默认 packaged frontend。
- 重复导入的限制只针对正式开局后的存档；在开局向导阶段，允许玩家重新导入/换源，用于调整小说结构、清洗文本和确认章节切分效果。

## UI Paradigm Recommendation

采用已确认的 **全屏 Chapter Setup Shell**：开局流程不是普通弹窗，也不是故事区里的卡片，而是正式游玩前的独立“章节建立”模式。

核心结构：

```text
[ 顶部：Opening Guide / 当前 save 状态 / 轻量说明 ]

[ 左侧：流程预览 Step Rail ]   [ 右侧：当前步骤主舞台 ]
                               [ 当前步骤说明 ]
                               [ 当前步骤内容/卡片/表单/对话区 ]
                               [ 底部：上一步 / 下一步 / 主行动 ]

[ 背景：弱化的书页/烛火/暗幕氛围，不露出正式剧情 composer ]
```

## Requirements

- `R1` 模式隔离：开局向导期间应使用独立 setup shell，隐藏正式游玩 composer，避免背后露出游戏界面或让玩家误以为已进入正式游玩。
- `R2` 流程预览：向导应展示稳定步骤列表，至少包括：导入小说、初始理解、角色设定、游玩倾向、开局确认。未实现步骤可显示为 locked/upcoming。
- `R3` 当前步骤舞台：当前步骤主内容应位于一个明确的 stage，而不是普通窄表单；导入步骤应先展示“粘贴文本”和“导入文件”两个方式卡片，卡片只写玩家可理解的适用场景，点击后再切换到对应输入界面。
- `R4` 导航范式：底部固定行动区应包含“上一步”“下一步/继续”“主行动”概念。当前只有第一步时，上一步可 disabled，导入前主行动为“导入小说”，导入成功后下一步显示为“继续初始化（后续）” disabled 或 upcoming。
- `R5` 后续可扩展：UI 结构应能容纳后续 architect 对话、候选角色卡片、角色草案确认和开局 brief 确认，不应为导入表单写死。
- `R6` 视觉方向：延续烛火书卷风格，但向导应更像“开局仪式/目录页”，不是普通后台表单；强调空间、层级和过渡感。
- `R7` 开发边界：只改 `apps/play-frontend-dev`，不改 `apps/platform-web/src/storage/default-frontend-files.ts`。
- `R8` 导入结果可视化：导入成功后应在导入步骤内展示轻量切分概览和章节卡片列表，让用户能确认章节数量、总字数和每章大致规模；不展示导入方式、识别置信度、算法说明、异常解释或冗长技术提示。
- `R9` 章节预览：切分信息展示界面顶部展示总览信息（书名、章节数、文本量）；下方采用左侧章节列表 + 右侧内嵌预览面板。点击章节卡片后右侧切换预览片段，用于快速判断切分是否合理；预览应读取已落盘章节内容或使用导入时生成的章节内容，不要求完整全文编辑。
- `R10` 重新导入边界：在开局向导未完成、尚未进入正式游玩前，导入完成态应提供重新导入入口，允许玩家换源或调整文本后覆盖当前 source corpus；正式开局后不提供重复导入入口。
- `R11` 覆盖提示：重新导入应明确提示会覆盖当前 source corpus 和章节索引；本任务不要求保留历史导入版本。
- `R12` 导入子流程：导入步骤内部状态至少包含“选择方式 → 对应输入 → 切分信息展示”；输入界面允许低调返回选择方式，导入成功后自动切换到切分信息展示界面。

## Acceptance Criteria

- [ ] 未导入 source 的 save 进入开发前端时显示全屏 setup shell，不显示正式游玩 composer。
- [ ] 页面有清晰流程预览，当前步骤“导入小说”高亮，后续步骤显示 upcoming/locked 状态。
- [ ] 导入步骤布局不再是单个普通表单框，而是具有明确标题区、说明区、导入方式区和结果区。
- [ ] 未选择导入方式时，导入步骤只展示“粘贴文本”和“导入文件”两个卡片按钮，每个卡片说明适用场景，不出现开发阶段术语或工程化设计决策。
- [ ] 点击“粘贴文本”后进入粘贴输入界面；点击“导入文件”后进入文件选择界面；两个输入界面不堆在同一屏。
- [ ] 输入界面提供低调的“更换导入方式”入口，可返回方式选择界面。
- [ ] 导入成功后自动进入切分信息展示界面，而不是只改变状态文字。
- [ ] 有明确底部行动区，包含 disabled 的“上一步”和主行动按钮；导入成功后显示下一步占位。
- [ ] 已导入 source 且仍处于开局向导阶段时，显示“小说已导入”状态、摘要、后续初始化占位，并提供明确的重新导入/换源入口。
- [ ] 重新导入入口会提示覆盖当前导入结果，并在确认后回到导入输入态或直接执行新的导入流程。
- [ ] 导入成功后进入切分信息展示界面，顶部显示轻量总览信息，只包含书名、章节数和总字数/文本量。
- [ ] 导入成功后显示章节卡片列表，至少包含章节序号、章节名和可读的字数/规模信息。
- [ ] 切分信息展示界面采用章节列表 + 右侧内嵌预览面板；默认预览第一章，点击章节卡片后切换预览。
- [ ] 预览只展示章节开头片段，用于验证切分效果，不展开整章全文、不使用 modal 弹窗。
- [ ] 代码结构能让后续步骤复用 setup shell，不需要每个步骤重新实现一套布局。
- [ ] `npm run build --workspace play-frontend-dev` 通过。

## Out of Scope

- 开局动画。
- architect Agent 对话、原著/原创角色完整初始化。
- 角色卡片渲染、角色草案落盘、开局剧情生成。
- 默认 packaged frontend 替换或维护。
- 导入后的章节编辑、手动重新切分、删除 source、导入版本历史。
- 正式开局后的重复导入入口。
