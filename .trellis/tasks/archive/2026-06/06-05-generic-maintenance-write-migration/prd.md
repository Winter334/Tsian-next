# Generic Maintenance Write Migration

## Goal

Move the default AIRP maintenance write path from the legacy
`MaintenancePatchDocument` / `apply-patch` compatibility path toward generic,
schema-validated `MemoryWriteOperation` writes, so the default runtime memory
update flow is represented by the same workflow primitives that players and mod
authors can inspect, edit, and eventually reuse for custom AIRP memory systems.

## What I Already Know

* The broader product goal is a strong workflow system that can cover AIRP core
  information processing instead of leaving important behavior hardcoded in the
  platform host.
* The previous task connected `memory-write` to `@tsian/memory-core` validation
  for built-in AIRP schema-covered targets while leaving custom
  namespace/collection writes storage-only.
* `defaultAirpMemorySchema` covers runtime `events`, `archives`, and `globals`
  under the built-in `airp` namespace.
* `apps/platform-web/src/workflow-host/default-workflow.ts` still routes
  `maintenance.patch` into an explicit `apply-patch` node.
* `builtin.maintenance` currently instructs the maintenance AI to output
  `MaintenancePatchDocument` JSON with `currentTime`, `globals`, `events`, and
  `archives`.
* `apply-patch` calls `applyMaintenancePatch()`, which still owns important
  compatibility behavior: runtime `currentTime` / `globals` updates, archive
  weak-name to strong-id resolution, event weak-name to archive-id resolution,
  legacy event/archive storage writes, and checkpoint creation.
* `memory-write` currently writes `MemoryWriteOperation[]` into save-scoped
  `memoryRecords`, creates checkpoints with events/archives/memoryRecords, and
  returns generic write outputs.
* The current generic memory storage does not migrate default event/archive
  tables; it writes records in `localDb.memoryRecords`.
* `memory-query(source: "event-archive")` calls `assembleRetrievalContext()` with
  `ctx.events`, `ctx.activeEvents`, `ctx.archives`, `ctx.globals`, and
  `ctx.currentTime`; those context fields are currently prepared by
  platform-host from legacy event/archive tables plus runtime snapshot state.
* New saves currently seed initial events and archives into legacy
  `localDb.events` / `localDb.archives`, while initial checkpoints include an
  empty `memoryRecords` slice unless other code has written records.
* The local DB is still prototype-versioned (`tsian-local-v9`), and the project
  convention allows clearing/reseeding prototype local data instead of writing
  compatibility migrations.

## Assumptions (Temporary)

* This task should be a narrow runtime migration slice, not the full replacement
  of retrieval, event/archive storage, schema resources, or workflow composition.
* `apply-patch` should remain available as a compatibility node while the
  default path starts moving toward `memory-write`.
* Name-reference resolution and checkpoint consistency remain platform-owned
  responsibilities.
* The current maintenance prompt is prototype/test scaffolding, not a stable
  contract that should drive architecture decisions.
* Prompt tuning is a required future workflow either way, so this task may
  change the maintenance output shape directly when that produces a cleaner
  generic workflow path.

## Requirements (Evolving)

* The default maintenance write path should move closer to generic
  `MemoryWriteOperation` without expanding `apply-patch` into the generic memory
  model.
* The migration must preserve current visible AIRP maintenance behavior for
  currentTime, globals, events, archives, strong reference resolution, and
  checkpoints unless explicitly scoped out.
* Schema-covered built-in AIRP memory operations must pass through the existing
  `memory-write` validation boundary when this task uses that boundary.
* Default retrieval should read the generic AIRP `memoryRecords` collections
  for this MVP so maintenance writes are consumed by the next turn's information
  flow.
* New default saves should provide initial default AIRP memory records for
  `events`, `archives`, and `globals`, not only legacy event/archive rows.
* New default saves may still seed legacy event/archive rows during this slice,
  but those rows are compatibility data for existing peripheral UI/bridge/debug
  paths, not the default AIRP core loop authority.
* `currentTime` should also be represented in generic AIRP memory as a reserved
  `airp/globals` record, so retrieval can derive narrative time from the same
  memory source as other default runtime state.
* This MVP uses weak-name relations (`entityTags`, archive `name`/`aliases`,
  and `linkedNames`) as the default relation surface. Strong id fields remain
  optional and may be generated by a later resolver/schema-relation slice.
* Workflow trace/debug output should make the maintenance write path inspectable.
* Custom memory experimentation must remain unblocked.

## Expansion Sweep

### Future Evolution

* Maintenance can later output generic operations directly, with editor forms
  derived from memory schema resources.
* A future workflow block/subgraph could package the whole AIRP maintenance
  chain as a reusable preset instead of exposing every low-level transform.

### Related Scenarios

* Retrieval still uses a special `event-archive` branch and will need a later
  decomposition step.
* Bridge APIs such as `applyPatch()` and `updateGlobals()` still need the legacy
  compatibility path until their callers migrate.

### Failure and Edge Cases

* Invalid operations should fail loudly before storage writes when the built-in
  AIRP schema covers the target.
* Partial side effects must follow the current checkpoint/fail-loud model; this
  task should not accidentally imply workflow-wide storage transactions.
* Changing maintenance output format directly risks AI output regressions, so
  adapter-based migration may be safer for the first slice.

## Candidate MVP Approaches

## Risk Clarification

The recommendation for Approach A is not primarily about protecting production
data or avoiding all visible behavior changes. The project is still in
prototype development, so breaking local prototype data or reseeding built-ins
is acceptable when the architecture becomes cleaner.

The risks Approach A tries to avoid are narrower:

* Losing a deterministic comparison point while changing both AI output shape
  and write semantics at the same time.
* Hiding name-reference resolution decisions inside prompt wording before those
  decisions are modeled as workflow/platform behavior.
* Making debugging harder if invalid memory operations appear, because it would
  be unclear whether the failure came from prompt generation, schema conversion,
  storage semantics, or workflow wiring.
* Replacing a compatibility layer with a prompt-only contract before we have a
  small operation-generation test surface.

Because the current prompt is only a prototype/test prompt and prompt tuning is
unavoidable, the user prefers architectural directness over preserving a
deterministic migration bridge for this slice.

### Approach A: Patch-To-Operations Adapter First

Keep the maintenance AI prompt output as `MaintenancePatchDocument` for now, add
a platform-owned adapter that converts the patch into default AIRP
`MemoryWriteOperation[]`, and route the default workflow through `memory-write`
for the schema-covered memory write portion where feasible.

**Pros**

* Preserves the mature maintenance prompt and current AI output contract.
* Creates a deterministic bridge from legacy patch shape to generic operations.
* Gives us focused tests for operation generation before asking the AI to emit a
  new shape directly.

**Cons**

* The default workflow may temporarily contain both compatibility and generic
  steps if currentTime/globals/event/archive legacy storage behavior cannot all
  move in one slice.
* The adapter can become another temporary compatibility layer that must be
  retired later.

### Approach B: Prompt Outputs MemoryWriteOperation Directly (Selected)

Change `builtin.maintenance` so the AI emits `MemoryWriteOperation[]` and change
the default workflow to feed that output into `memory-write`.

**Pros**

* The workflow graph becomes conceptually cleaner.
* The AI-facing contract aligns immediately with the generic memory model.

**Cons**

* Higher behavioral risk because the maintenance prompt format changes and may
  need more scenario regression coverage.
* Existing name-reference resolution currently lives in `applyMaintenancePatch()`;
  direct operations would need an explicit resolution stage or stricter prompt
  requirements.
* Some current legacy behaviors, especially runtime state updates and weak-name
  resolution, must either be explicitly preserved through platform behavior or
  consciously deferred.

### Approach C: Compatibility Mirror

Keep `apply-patch` as the authoritative write path for now, but mirror the
resulting event/archive/global writes into generic `memoryRecords` through
`MemoryWriteOperation` so default memory becomes queryable through the generic
collection path.

**Pros**

* Low risk to current gameplay behavior.
* Starts populating generic records for future retrieval/workflow experiments.

**Cons**

* Duplicates default memory state across legacy tables and generic records.
* Delays making `memory-write` the true default write authority.

## Decision (ADR-lite)

**Context**: The project now has generic schema-aware `memory-write`, but the
default AIRP maintenance chain still emits `MaintenancePatchDocument` and writes
through `apply-patch`.

**Decision**: Use Approach B for the MVP. Change the default maintenance AI
output contract toward `MemoryWriteOperation[]` and route the default workflow
through the schema-aware `memory-write` node rather than adding a patch-to-op
adapter first.

**Consequences**: This favors architectural clarity and faster exposure of the
real generic write path. It accepts prompt iteration and short-term behavior
regression risk as normal prototype work. The implementation must still decide
how much of the old `applyMaintenancePatch()` behavior, especially currentTime,
globals, and weak-reference resolution, belongs in this MVP.

### Default Retrieval Reads Generic AIRP Memory

**Context**: If maintenance writes `MemoryWriteOperation[]` into generic
`memoryRecords` while default retrieval still reads legacy event/archive tables,
the default AIRP loop would write memory that the next turn does not consume.

**Decision**: The MVP should also switch default retrieval to consume generic
AIRP memory records for the built-in `events`, `archives`, and `globals`
collections.

**Consequences**: This makes the slice larger than write migration alone, but it
keeps the default loop coherent and moves the core read/write memory path toward
the workflow system together.

### Initial Generic Memory Seeding Scope

**Context**: Switching retrieval to generic records means the first turn needs
generic `airp/events`, `airp/archives`, and `airp/globals` data. Current save
creation only seeds legacy event/archive tables and snapshot globals.

**Decision**: Seed both generic AIRP memory records and legacy event/archive
rows for new default saves. The default AIRP workflow should treat generic
memory records as authoritative, while legacy rows remain temporary compatibility
data for existing peripheral UI, bridge, and debug consumers.

**Consequences**: This keeps the core workflow migration focused while avoiding
a broad UI/bridge rewrite in the same slice. Follow-up work can retire or remap
legacy consumers once the generic default memory path is stable.

### CurrentTime Lives In Generic Globals

**Context**: Current narrative time is stored today on the runtime snapshot,
while generic AIRP memory has a `globals` collection that can represent
save-scoped runtime state as `{ key, value }` records. If retrieval reads generic
memory but still reads time from the snapshot, the default loop would keep a
split authority.

**Decision**: Represent `currentTime` as a reserved `airp/globals` record, for
example record id/key `currentTime` with data
`{ "key": "currentTime", "value": "YYYY-MM-DD HH:mm" }`. Default retrieval
should derive narrative time and global state from generic globals.

**Consequences**: Generic AIRP memory becomes the default narrative state truth
for retrieval/maintenance. Runtime snapshot fields may still exist as
compatibility/projection data during the transition, but they should not be the
default retrieval authority for currentTime/globals in this slice.

### Strong Reference Resolution Deferred

**Context**: The legacy patch applier resolves weak entity/archive names into
strong id fields (`entityArchiveIds`, `linkedArchiveIds`). Current retrieval
logic primarily uses weak-name fields (`entityTags`, archive `name`/`aliases`,
and `linkedNames`) for direct, present, event, and bridge recall.

**Decision**: The MVP should rely on weak-name relations. Maintenance may output
strong id fields if it can, but the generic default path must not require
strong-reference generation. A later resolver/schema-relation slice can add
deterministic strong id enrichment.

**Consequences**: This avoids reintroducing hidden relation mutation while the
core read/write memory authority moves to generic records. Some precision
benefits from strong ids are deferred.

## Out of Scope

* Removing `apply-patch` entirely.
* Replacing all legacy event/archive storage in one step unless explicitly
  chosen as the MVP.
* Building first-class editable memory schema resources.
* Fully decomposing `memory-query(source: "event-archive")` into separate
  ranking/filtering/relation/template nodes.
* Deterministic weak-name to strong-id relation resolver.
* Adding event/archive-specific workflow node types.
* Adding a broad workflow permission/capability system.

## Technical Approach

* Keep `memory-query(source: "event-archive")` as the high-level default AIRP
  retrieval strategy for this slice, but back its default context with generic
  `airp/events`, `airp/archives`, and `airp/globals` memory records instead of
  legacy event/archive tables.
* Add a platform-web helper that projects save-scoped generic AIRP memory records
  into the runtime shapes retrieval already understands:
  events, active events, archives, globals, and `currentTime`.
* Treat `airp/globals/currentTime` as a reserved global record. Retrieval should
  read `currentTime` from that record and build the normal globals map from the
  remaining `airp/globals` records.
* Seed new saves with generic AIRP memory records for initial events, archives,
  globals, and the reserved `currentTime` record. Continue seeding legacy
  event/archive rows in this slice for existing peripheral consumers.
* Update `builtin.maintenance` so the AI emits `{ "operations": [...] }` where
  each item is a `MemoryWriteOperation` targeting `events`, `archives`, or
  `globals`.
* Update the default workflow so maintenance output goes to a `memory-write`
  node instead of the `apply-patch` node.
* Leave `apply-patch` registered and available as a compatibility node, but do
  not use it in the default workflow's maintenance write path.
* Use weak-name relations as the default relation surface. Do not require
  `entityArchiveIds` or `linkedArchiveIds` to be generated in this slice.
* Keep custom namespace/collection writes unchanged.
