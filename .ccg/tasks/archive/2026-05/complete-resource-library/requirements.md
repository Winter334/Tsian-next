# 补全资源库功能 — Requirements

## 目标

补全 `ResourceLibraryView`，把当前占位页面推进到可实际管理平台级全局资源：

1. 提示词预设库（prompt presets）
2. 世界书库（world books）
3. 工作流预设库（workflow presets）

## 背景约束

- 平台架构已确认为“全局资源库 + 模组唯一入口 + 资源绑定链路”。
- 工作流预设与提示词预设同级，是全局库资源，不做模板/实例二分。
- 模组不直接管理提示词预设和世界书；工作流节点引用全局库资源。
- 原型期允许破坏性 IndexedDB/Dexie 调整，不做迁移兼容。
- UI 必须延续 Deep Phosphor：暗色工业风、霓虹青、零圆角。
- 当前资源库页面只有 tab + empty 占位，需要接入真实本地数据。

## 范围

本批优先补全“本地全局资源库管理闭环”：

- contracts：定义平台全局资源类型与最小字段。
- storage：提供 IndexedDB/Dexie 本地 CRUD。
- ResourceLibraryView：展示列表、创建、编辑、删除/复制等最小可用操作。
- 工作流预设资源应能保存 `WorkflowDefinition`。
- 提示词预设/世界书可以先用结构化 JSON 或最小字段编辑，避免过度设计复杂编辑器。

## 非目标

- 不做官方后端同步。
- 不做账号/权限。
- 不做资源打包导入导出完整流程。
- 不做提示词/世界书的高级 SillyTavern 全量编辑器。
- 不重做工作流画布本身。

## 验收标准

- `/resources` 不再只是占位，三个 tab 都能显示真实资源数量和列表。
- 可以创建、编辑、删除至少一种资源；三类资源采用一致的最小 CRUD 模式。
- 数据刷新后仍保存在本地 IndexedDB。
- 类型定义与存储函数清晰，不靠散落的 `any` 隐式协议。
- 不引入大而泛的抽象框架。

## 需求完整性评分

8/10

- 目标明确：3/3
- 预期结果：2/3（具体字段可在规划中定）
- 边界范围：2/2
- 约束条件：1/2（删除确认/导入导出细节待定）
