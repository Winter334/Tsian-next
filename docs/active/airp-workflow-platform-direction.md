# AIRP Workflow Platform Direction

## 1. 文档目的

本文档记录 2026-05-29 已经确认的产品方向与近期实施取舍，避免后续会话只保留零散口头结论。

它不是具体代码设计文档，而是回答三个问题：

- Tsian 到底要成为哪类系统
- AIRP 为什么适合抽象成工作流
- 近期哪些工作应该先做，哪些扩展应该延后

## 2. 核心定位

Tsian 的核心定位不是单纯的 AI 角色扮演内容应用，而是一套 AIRP 工作流编辑和运行平台。

项目的核心理念是：

`AIRP 可以抽象成一个可预设、可组合、可调试的 AI 文本处理流程。`

因此，平台最重要的能力不是替玩家固定一条官方剧情链路，而是让玩家和作者能够围绕自己的 AIRP 需求组合出合适的工作流。

## 3. AIRP 回合抽象

一轮 AIRP 可以抽象为：

`玩家输入 -> 上下文收集 -> 记忆查询 -> prompt 组合 -> 主 AI 调用 -> 输出解析 -> 维护/规则处理 -> 状态写回 -> 调试追踪`

这个抽象解释了当前各系统的定位：

- prompt preset 是工作流中的提示词资源
- world book 是工作流中的上下文资源
- memory 是工作流中的查询、组织、维护和写回阶段
- runtime state 是工作流读写的运行目标
- mod 是资源包加默认工作流预设
- debug 是节点级输入、输出和副作用的可见性

## 4. 工作流系统的主轴地位

后续设计应优先围绕工作流基础设施收敛，而不是把每种 AIRP 能力固化成平台硬编码功能。

这意味着：

- 用户应能选择当前存档使用哪个 workflow preset
- 节点之间的数据传递应从纯字符串变量名，逐步升级为可理解的输入/输出语义槽
- 工作流运行过程应能被追踪、复盘和调试
- 当前平台预设能力应逐步以节点或资源的方式显性化
- 平台应保留安全边界，例如存储、checkpoint、回滚、迁移和最终状态写入

工作流不是附加编辑器，而是 AIRP 系统能力的主组织方式。

## 5. 记忆系统方向

当前不实现第二套记忆系统。

近阶段目标是先把现有事件/档案记忆系统打磨好，并让它在工作流中变得可见、可配置、可诊断。

原因：

- 当前事件/档案体系已经和运行时状态、维护 patch、检索调试有较深绑定
- 直接新增一套摘要+关键词记忆系统会分散主链打磨
- 玩家真正需要的第一步不是多套架构并存，而是能理解、调整和调试当前记忆链路
- 工作流基础设施补全后，未来再扩展新记忆架构会更稳

长期应保持概念边界开放：

- 当前事件/档案系统是官方参考实现，不是唯一可能的记忆模型
- 未来可以引入 summary+keywords、向量片段、规则型状态表等其他记忆存储
- 不应把未来所有记忆架构都强行塞进事件/档案内核

但近期不要为了这种可能性提前设计完整 MemoryStore 插件平台。

## 6. Compute 节点定位

compute 节点可以作为高级逃生口，让高级作者实现临时转换、评分、筛选和局部记忆实验。

但它不应成为普通用户配置 AIRP 的主要方式。

如果某项能力对 AIRP 主链很关键，应优先沉淀为明确节点、资源类型或平台能力，而不是要求用户通过 compute 自己复刻整套平台逻辑。

## 7. 平台应保留的安全边界

即使工作流能力增强，以下部分仍应由平台拥有：

- 本地存储结构和迁移
- checkpoint 和回滚
- 事务一致性
- 运行时状态写入校验
- mod 控制工作流与平台控制写回能力的权限边界
- 缺失资源、非法节点和危险写回的失败策略

玩家可以自定义 AIRP 流程，但不应该被迫承担底层状态一致性的风险。

## 8. 近期实施路线

推荐近期顺序：

1. 存档级 workflow preset override
2. 节点输入/输出 schema 与语义槽选择
3. workflow run trace 和调试 UI
4. 将当前记忆链条逐步节点化，包括 retrieval、memory compose、maintenance、patch normalize、apply patch
5. 在基础设施稳定后，再评估通用 MemoryStore 和第二类记忆架构

其中第一步最关键，因为它让 workflow preset 从资源库里的可编辑对象，正式变成每个存档可选择的运行策略。

## 9. 设计取舍

当前确认的取舍是：

- 先打磨一套官方事件/档案记忆系统，不并行维护两套记忆架构
- 先补工作流基础设施，不急着做庞大的插件平台
- 先让平台预设链路可视化、可配置、可调试，再追求完全开放
- 先保证存档和 mod 能稳定选择工作流，再做更细节点能力
- 缺失引用和危险能力应明确失败，不做静默 fallback

这条路线能保留最终平台化潜力，同时避免在基础设施不足时过早抽象。
