# AIRP Patch Contract Skeleton

## 1. 文档目的

本文档用于确定项目的`状态修改契约骨架`。

当前阶段只回答以下问题：

- 维护 AI 应输出什么形态的修改结果
- 这些修改结果在高层上如何分组
- 哪些动作当前保留，哪些动作当前不引入

当前阶段明确不讨论：

- 具体数据库写入细节
- 具体提示词逐句写法
- 复杂回滚与审计
- 多事件并行维护策略

## 2. 当前结论

当前推荐的 patch 契约强调：

- 保持 JSON 结构可读
- 统一修改动作语义
- 不引入过度通用的底层指令系统
- 不要求所有领域对象共享同一套字段集合

可以压缩为一句话：

`统一修改动作，不统一字段全集。`

## 3. 顶层分组

当前推荐 patch 文档按以下顶层分组：

1. `currentTime`
2. `globals`
3. `events`
4. `archives`

其中：

- `currentTime` 表示当前叙事时间
- `globals` 表示不属于单个实体的全局状态
- `events` 表示事件记录修改
- `archives` 表示实体档案修改

当前不再额外暴露一个名为 `runtime` 的顶层概念。

## 4. 当前推荐动作语义

当前推荐统一使用：

- `target`
- `set`
- `create`

### 4.1 target

`target` 表示“改谁”。

它的职责是：

- 在已有对象中定位修改目标

### 4.2 set

`set` 表示“把这些字段改成什么”。

它的职责是：

- 新增字段值
- 覆盖已有字段值

补充口径如下：

- `set` 当前只要求支持顶层字段替换
- 不做深层路径 patch
- 若某个字段值本身是对象，则维护 AI 直接输出该字段的新完整对象

### 4.3 create

`create` 表示“新建一个对象”。

它当前主要用于：

- 新建档案

当前不要求所有分组都立即支持 `create`。

## 5. 当前不引入的动作

当前不引入：

- `del`

原因如下：

- 当前系统希望字段结构整体稳定
- 前端包只渲染约定好的字段
- AI 不应自由增删字段集合
- 删除动作会让结构存在性变得更不稳定

因此当前固定以下口径：

- 缺字段 = 不改
- 要表达“当前无值”，使用该字段约定的空态
- 空态由对应字段定义决定，而不是靠删除字段表达

## 6. globals 的口径

`globals` 用于承载不属于单个实体的状态。

例如：

- 天气
- 当前章节名
- 全局公式字符串
- 全局样式片段

当前推荐口径为：

- `globals` 只需要 `set`
- 平台不理解这些值的具体语义
- 平台只负责存储、注入、返回
- 前端包自行决定如何消费这些值

## 7. archives 的口径

`archives` 用于修改实体档案。

当前进一步固定如下：

- 档案既包含公共基础字段，也包含扩展字段
- 扩展字段不在外层 patch 里再单独拆成一组“实体运行时状态”
- 修改实体状态，就是修改档案

因此：

- 更新已有档案时，使用 `target + set`
- 创建新档案时，使用 `create`

补充口径如下：

- `set` 中既可以出现基础字段，也可以出现扩展字段
- 扩展字段是否允许出现，由该档案的 `kind` 对应字段定义决定
- 平台只校验字段是否允许写入，不负责理解字段内容语义

## 8. events 的口径

`events` 用于修改事件记录。

当前建议继续保持最小口径：

- 以当前进行中的事件为主要修改目标
- 当前仍以 `target = active` 为核心

也就是说：

- 当前不为未来可能出现的多事件维护提前设计复杂事件定位系统
- 当前事件 patch 仍然服务于最小闭环

## 9. kind 与字段定义的关系

当前结论是：

- patch 协议统一
- 字段集合不统一

真正决定“某个实体允许有哪些扩展字段”的，不是 patch 本身，而是：

- 档案上的 `kind`
- `kind` 对应的字段定义

这意味着：

- 平台不为所有实体维护一套统一的大字段全集
- 不同 `kind` 可以有不同扩展字段
- 同一基础类下的不同变体，也可以有不同字段集合

## 10. 当前推荐口径

截至目前，patch 契约的推荐口径可以压缩为：

`patch 顶层当前分为 currentTime / globals / events / archives 四组；统一采用 target / set / create 这套动作语义；缺字段表示不改，不引入 del；全局状态用 globals 承载，实体状态直接作为档案的一部分通过 archives 修改，不再额外暴露实体运行时状态层；不同实体允许写入哪些扩展字段，由其 kind 对应的字段定义约束。`

## 11. 后续设计入口

在 patch 契约骨架已经收敛后，后续可以继续围绕这些主题展开：

1. `kind` 对应字段定义的最小结构
2. `globals` 字段值允许到什么 JSON 范围
3. 事件 patch 在多进行中事件条件下的定位策略
4. 平台对非法字段的处理口径
