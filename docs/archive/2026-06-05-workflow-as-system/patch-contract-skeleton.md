# AIRP Patch Contract Skeleton

## 1. 文档目的

本文档用于确定项目的`状态修改契约骨架`。

当前阶段只回答以下问题：

- 维护 AI 应输出什么形态的修改结果
- 这些修改结果在高层上如何分组
- 哪些动作当前保留，哪些动作当前不引入

当前阶段明确不讨论：

- 具体数据库写入细节
- 具体提示词逐句写法
- 复杂回滚与审计
- 多事件并行维护策略

## 2. 当前结论

当前推荐的 patch 契约强调：

- 保持 JSON 结构可读
- 统一修改动作语义
- 不引入过度通用的底层指令系统
- 不要求所有领域对象共享同一套字段集合

可以压缩为一句话：

`统一修改动作，不统一字段全集。`

## 3. 顶层分组

当前推荐 patch 文档按以下顶层分组：

1. `currentTime`
2. `globals`
3. `events`
4. `archives`

其中：

- `currentTime` 表示当前叙事时间
- `currentTime` 当前使用 `YYYY-MM-DD HH:mm` 字符串，例如 `2026-04-27 13:58`，不使用 ISO、时区或秒
- `globals` 表示不属于单个实体的全局状态
- `events` 表示事件记录修改
- `archives` 表示实体档案修改

当前不再额外暴露一个名为 `runtime` 的顶层概念。

## 4. 当前推荐动作语义

当前推荐统一使用：

- `target`
- `set`
- `create`

### 4.1 target

`target` 表示“改谁”。

它的职责是：

- 在已有对象中定位修改目标

### 4.2 set

`set` 表示“把这些字段改成什么”。

它的职责是：

- 新增字段值
- 覆盖已有字段值

补充口径如下：

- `set` 当前只要求支持顶层字段替换
- 不做深层路径 patch
- 若某个字段值本身是对象，则维护 AI 直接输出该字段的新完整对象

### 4.3 create

`create` 表示“新建一个对象”。

它当前主要用于：

- 新建档案
- 新建事件记录

当前不要求所有分组都支持复杂定位或批量创建；只在已经确认需要的分组中使用显式 `create`，不使用“无 target 的 set 等价 create”这类隐式语义。

## 5. 当前不引入的动作

当前不引入：

- `del`

原因如下：

- 当前系统希望字段结构整体稳定
- 前端包只渲染约定好的字段
- AI 不应自由增删字段集合
- 删除动作会让结构存在性变得更不稳定

因此当前固定以下口径：

- 缺字段 = 不改
- 要表达“当前无值”，使用该字段约定的空态
- 空态由对应字段定义决定，而不是靠删除字段表达

## 6. globals 的口径

`globals` 用于承载不属于单个实体的状态。

例如：

- 天气
- 当前章节名
- 全局公式字符串
- 全局样式片段

当前推荐口径为：

- `globals` 只需要 `set`
- 平台不理解这些值的具体语义
- 平台只负责存储、注入、返回
- 前端包自行决定如何消费这些值

补充固定边界如下：

- 非 AI 渠道如果需要修改运行时状态，也必须走平台正式写入口
- 前端包不能绕过平台直接写底层存储
- 这样才能保证 checkpoint、回溯、调试视图与真实运行时状态保持一致

## 7. archives 的口径

`archives` 用于修改实体档案。

当前进一步固定如下：

- 档案既包含公共基础字段，也包含扩展字段
- 扩展字段不在外层 patch 里再单独拆成一组“实体运行时状态”
- 修改实体状态，就是修改档案

因此：

- 更新已有档案时，使用 `target + set`
- 创建新档案时，使用 `create`

补充口径如下：

- `set` 中既可以出现基础字段，也可以出现扩展字段
- 扩展字段是否建议出现，由该档案的 `type` 对应最终字段定义决定
- 平台只校验字段是否允许写入，不负责理解字段内容语义
- `name` 是叙事可读标签，不是唯一键；程序唯一标识是 `id`
- 同名档案允许存在，不为了实体管理限制正文 AI 命名
- 档案创建使用显式 `create`，不做程序层名称去重或自动合并
- 创建时允许最小创建，后续再用 `target + set` 补齐；不要为了填满类型字段而编造正文没有支持的内容

## 8. events 的口径

`events` 用于修改事件记录。

当前建议从单一 active event 升级为轻量多 active event：

- 维护 AI 可以更新已有事件：`target + set`
- 维护 AI 可以创建新事件：`create`
- 系统允许暂时没有 active event
- 系统允许同时存在多个 `ongoing` 事件
- 当剧情判断某个进行中事件完成时，维护 AI 将其 `status` 改为 `done`
- 当新事件产生时，维护 AI 用 `create` 新建一个 `ongoing` 事件

也就是说：

- 当前补齐多 active 的最小能力，但不做复杂事件调度器
- 不做事件删除、不做事件分支、不做复杂事件依赖图
- `target = active` 不再作为长期唯一核心；后续应优先使用具体事件 id 定位已有事件

事件创建示例：

```json
{
  "create": {
    "status": "ongoing",
    "time": "2026-04-27 21:09",
    "entityTags": ["叶临", "韩照"],
    "content": "2026年4月27日 21:09，叶临开始追问韩照二更前后的盐仓行踪。"
  }
}
```

事件更新示例：

```json
{
  "target": "event id",
  "set": {
    "status": "done",
    "time": "2026-04-27 21:16",
    "content": "2026年4月27日 21:16，韩照承认二更前后陪罗缇进过盐仓，盐仓钥匙线索暂告一段落。"
  }
}
```

补充边界如下：

- patch 中的 `events` 只修改运行时事实事件
- 模组静态层中的 `event catalog` 不属于 patch 的 `events` 组
- 预设事件被注入给正文 AI，不等于已经发生，不直接生成运行时事件 patch
- 只有正文实际产生相关事实后，维护 AI 才通过 `events.create` 或 `target + set` 将其落成运行时事件
- 当前不在运行时事件中增加 `sourceCatalogEventId` 之类的显式来源追踪字段

## 9. 关系字段与内部强引用

`entityTags` 和 `linkedNames` 继续作为 AI 可读的叙事名称关系：

- `entityTags` 表示事件正文涉及的实体名称
- `linkedNames` 表示档案与其他实体的稳定强关联名称

由于 `name` 不是唯一键，名称关系可能存在歧义。当前固定以下口径：

- 名称关系永远保留，用于 AI 阅读、UI 展示和弱检索
- 程序内部可以维护 id 强关系，例如事件上的 `entityArchiveIds`、档案上的 `linkedArchiveIds`
- 维护 AI 默认仍只输出 `entityTags / linkedNames`，不额外承担 id 绑定负担
- 平台应用 patch 时，只在当前明确可见实体池中唯一解析名称时写入内部 id 强关系
- 如果名称同名且无法唯一解析，宁愿不写内部 id 强关系，也不猜测绑定
- 当前不做全库扫描补齐、不做后续自动回填、不做程序猜测修复
- 少数强关系缺失留给未来玩家手动修复记忆图

当前明确可见实体池包括：

- 本轮维护 AI 可见的档案
- 当前 active event 已有关联档案
- 本轮 patch 明确 target 更新的档案
- 本轮 patch create 新建的档案

## 10. type 与字段定义的关系

当前结论是：

- patch 协议统一
- 字段集合不统一

真正决定“某个实体应有哪些扩展字段”的，不是 patch 本身，而是：

- 档案上的 `type`
- `type` 对应的最终字段定义

这里进一步固定以下口径：

- 程序内部可以用父类 / 子类式类型定义复用字段
- AI 不需要看到内部继承关系
- 给 AI 的 patch 提示应展示该 `type` 合并后的最终合法字段列表
- 存储对象仍然保持扁平，不按类型继承结构嵌套保存

这意味着：

- 平台不为所有实体维护一套统一的大字段全集
- 不同 `type` 可以有不同扩展字段
- 内部父类字段可以被多个 `type` 复用，但最终写入档案时仍是扁平字段

## 11. 当前推荐口径

截至目前，patch 契约的推荐口径可以压缩为：

`patch 顶层当前分为 currentTime / globals / events / archives 四组；统一采用 target / set / create 这套动作语义；缺字段表示不改，不引入 del；全局状态用 globals 承载，实体状态直接作为档案的一部分通过 archives 修改，不再额外暴露实体运行时状态层；events 支持轻量多 active 与显式 create；name 是叙事标签而非唯一键；entityTags / linkedNames 是 AI 可读名称关系，内部 id 强关系只在当前可见实体池能唯一解析时写入；不同实体建议写入哪些扩展字段，由其 type 对应最终字段定义约束；内部类型继承只用于复用字段定义，不进入 AI patch 结构和存储结构。`

## 12. 后续设计入口

在 patch 契约骨架已经收敛后，后续可以继续围绕这些主题展开：

1. `type` 对应字段定义的最小结构
2. `globals` 字段值允许到什么 JSON 范围
3. 事件 patch 在多进行中事件条件下的定位策略
4. 平台对非法字段的处理口径
