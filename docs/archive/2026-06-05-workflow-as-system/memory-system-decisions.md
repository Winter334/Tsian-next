# AIRP Memory System Decisions

## 1. 文档目的

本文档用于记录当前已经收敛的`记忆系统设计决策`。

它不展开具体实现，不讨论数据库、索引细节或提示词逐句写法，只记录当前确认的方向与边界。

## 2. 总体结论

当前记忆系统采用以下总方向：

- 主存储仍以`事件`为核心
- 事件采用更轻的`事件记录`形态，而不是沉重的展示卡
- `档案`作为辅助层，承载叙事实体信息
- `原文证据`作为冷存储，仅在必要时回拉

可以压缩成一句话：

`事件记录为主，实体档案为辅，原文证据冷存储。`

## 3. 为什么继续采用事件

当前结论是：不回到每轮摘要，也不回到原文片段直存。

原因如下：

- 原文片段细节多，但噪声高、token 高，长期检索会越来越脏
- 每轮摘要 token 低，但细节损失过快，不适合长剧情连续性
- 事件天然更接近 AIRP 里真正需要记住的内容，即“发生过什么”
- 事件比原文更短、比摘要更稳、比事实图更具叙事可读性

因此，事件仍然是当前最合理的主记忆单元。

## 4. 事件记录与档案的关系

### 4.1 事件记录

事件记录负责承载：

- 某段剧情中发生了什么
- 它发生在什么时间锚点
- 它目前属于什么状态
- 它关联了哪些叙事实体

事件记录是叙事连续性的主骨架。

当前事件维护进一步固定为轻量多 active 口径：

- 维护 AI 持续更新仍在进行的 active event
- 当剧情判断某个 active event 完成时，将其 `status` 改为 `done`
- 系统允许暂时没有 active event
- 当新事件产生时，维护 AI 创建新的 `ongoing` event
- 系统允许同时存在多个 `ongoing` event，但当前不做复杂事件调度器、事件分支或事件依赖图

当前进一步收敛的口径是：

- 事件记录同时保留 `结构标签` 与 `自然语言正文`
- 结构标签主要用于检索、消歧和实体注入辅助
- 自然语言正文主要用于直接注入与阅读
- 事件结构标签当前只使用`实际存在的叙事实体主名称`
- 不使用抽象概念词作为事件标签

补充关系口径如下：

- `entityTags` 是 AI 可读的实体名称标签，不是程序唯一关系键
- 实体名称允许同名，正文 AI 不应为了实体管理被限制命名自由
- 程序内部可以为事件维护 `entityArchiveIds` 这类 id 强关系
- 内部 id 强关系只在当前可见实体池中能唯一解析名称时建立
- 如果同名导致无法唯一解析，保留名称弱关系，不猜测绑定，不做自动补齐

因此，事件正文本身默认应包含足以锚定该事件的：

- 时间
- 地点
- 人物

但表达方式应保持自然语言，不要求机械模板化。

补充固定以下边界：

- `currentTime` 和事件记录上的 `time` 是结构化时间锚点，主要服务记忆系统、排序、检索、消歧、注入和前端渲染
- 结构化时间当前采用 `YYYY-MM-DD HH:mm` 字符串，例如 `2026-04-27 13:58`；它是叙事世界时间，不使用 ISO、时区或秒
- 公共时间状态用于帮助系统串联剧情线，避免长期记忆只能依赖语义和事件先后关系推断时间线
- 没有公共时间并不意味着无法游玩；它是记忆系统增强能力，不是剧情自然语言的替代品
- 玩家输入、正文 AI 输出、事件正文、档案正文都应保持世界内自然语言表达
- 事件正文可以在有助于注入理解时包含自然语言时间、地点、人物信息，但不强制机械写入结构化时间
- 程序不应自动改写玩家输入、正文 AI 输出或维护 AI 写出的事件正文来适配时间字段
- 前端可以把结构化时间渲染成世界观历法，但渲染结果不应反向污染底层剧情正文
- AI 注入使用“视图时间”：平台默认从结构化时间生成普通可读文本；游玩前端可以在交互请求中传入已按自身世界观处理的格式化时间
- 正文 AI、检索相关文本、维护 AI 都可以看到格式化时间；维护 AI 同时看到结构化时间，结构化时间用于输出 `currentTime` / `event.time`，格式化时间只用于事件正文、档案正文等自然语言表达
- 格式化时间不是存储字段，不新增 `display` / `currentTimeDisplay`；底层仍只保存结构化的 `currentTime` 和事件 `time`
- 档案字段只写世界内事实，不写“已记录在时间线中”等系统实现说明

### 4.2 实体档案

实体档案负责承载：

- 人物
- 地点
- 关键物件
- 组织等可复用叙事实体

档案不负责承载完整剧情，只负责补充实体侧信息。

档案关系进一步固定为：

- `name` 和 `aliases` 是叙事标签，不要求全局唯一
- `id` 才是程序层唯一标识
- `linkedNames` 是 AI 可读名称关系，不是程序唯一关系键
- 程序内部可以为档案维护 `linkedArchiveIds` 这类 id 强关系
- 维护 AI 默认不需要输出 id 关系字段；平台只在当前可见实体池唯一解析时写入内部强关系
- 当前不做全库扫描补齐、不做后续自动回填、不做程序猜测修复
- 少数强关系缺失留给未来玩家手动修复记忆图
- 新档案使用显式 `create`，不做名称去重或自动合并
- 档案允许最小创建，后续再随剧情补齐；不要为了字段完整而编造正文没有支持的内容

### 4.3 原文证据

原文证据不作为主注入内容。

它的职责只有一个：

- 当事件记录不足以支撑回答或需要核对细节时，作为冷存储被回拉

## 5. 关键物件是否进入档案层

当前结论是：

`程序不判断物件是否重要，是否入档案由维护 AI 的提示词和沉淀策略决定。`

程序层只处理“已经进入档案层的叙事实体”，不负责判断某个物件够不够重要。

因此：

- 重要物件可以进入档案层
- 不是所有剧情中出现的物件都必须进入档案层
- 这项判断权不放在程序逻辑里

## 6. 时间在检索中的地位

当前结论是：

`时间不是默认主召回轴，而是精确锚定、排序和消歧轴。`

也就是说：

- 大多数实际游玩情况下，历史事件召回默认依赖非时间要素
- 只有在玩家明确回忆历史时，时间才作为重要消歧条件参与
- 当前系统时间通常只和当前进行中的事件强相关，不应默认拿来检索历史事件

## 7. 记忆检索 v2 方向

当前进一步收敛为“双轨检索”：

- 默认结构检索：以实体图和事件时间线为主，稳定维护剧情连续性
- 可选 AI 增强检索：以语义关键词和向量缓存为补充，召回结构检索覆盖不到的相似记忆

默认结构检索是基础链路；AI 增强检索只作为可选增强，不接管基础记忆链路。

### 7.1 当前实体池

检索前先构建当前实体池。

无条件注入的实体档案包括：

- 直接命中实体：玩家输入和最近上下文中明确出现的实体
- 当前在场实体：当前场景中应被正文 AI 视为在场或关键的实体，初期可由 active event 的 `entityTags` 承担
- 事件命中实体：最终注入事件的 `entityTags` 对应实体

这些实体不是普通检索扩展项，而是当前上下文理解所需的基础档案。

### 7.2 关联实体的定位

关联实体不默认注入。

关联实体的主要价值是：

- 作为事件检索时的弱信号
- 作为直接实体与历史事件之间的桥接实体
- 为剧情连续性提供候选，而不是解释当前上下文中已经出现的概念

因此当前口径是：

- 只做一跳关联，不做递归扩散
- 不因为 `linkedNames` 存在就默认注入关联档案
- 关联实体只有在成为桥接实体、共同关联实体、当前在场实体，或出现在最终注入事件中时，才进入注入候选

### 7.3 默认结构检索

默认结构检索不再依赖检索 AI 生成实体查询词组作为主链路。

基础流程是：

1. 使用直接命中实体、当前在场实体、active event 实体构成高权重实体池
2. 可引入少量一跳关联实体作为低权重桥接候选
3. 遍历事件池，按事件 `entityTags` 与实体池的匹配情况计算候选分
4. 选出少量高质量 seed event
5. 以 seed event 为中心，按时间线补充相邻事件，形成事件链片段

高质量 seed event 的判断方向：

- 同时命中多个当前实体
- 命中直接实体或当前在场实体
- 与 active event 的 `entityTags` 高度重合
- 命中更稀有、更能定位线索的实体
- 能解释当前实体的 `situation / focus` 或当前事件为何发展到这里
- 能与相邻事件组成连续片段，而不是孤立碎片

### 7.4 seed event 数量

seed event 数量应可配置。

初始阶段可以比保守方案更激进，因为现代大模型上下文较大；如果实际测试发现噪声或注意力分散，再降低数量。

默认结构检索不依赖简单玩家意图关键词来决定 seed 数量。

更推荐的默认调节依据是：

- 当前命中实体数量
- 当前在场实体数量
- active event 涉及实体数量
- 高质量候选事件是否跨越多条线索

实体牵涉越复杂，可以允许更多 seed event；但每个 seed event 仍必须过质量阈值。

### 7.5 相邻事件链片段

历史事件注入不追求散点数量，而优先追求连续性。

找到 seed event 后，可以注入它在时间线上的相邻事件，形成小段事件链。

这样做的目的：

- 给正文 AI 提供更连续的因果脉络
- 避免注入一组语义相似但彼此割裂的事件
- 让事件记录真正承担“剧情枝干”的作用

相邻事件数量也应可配置。

### 7.6 AI 增强检索

检索 AI 不移除，但改为可选的 AI 增强检索。

开启后，检索 AI 不再负责生成实体查询词组，而是根据当前上下文和玩家输入生成非实体语义关键词或检索意图。

这类关键词用于传统 RAG 风格的向量检索，例如：

- 勒索
- 伪证
- 钥匙交接
- 妖气遮蔽
- 尸体搬运

AI 增强检索的结果作为额外候选来源：

- 可检索事件记录
- 可检索实体档案
- 只保留相似度阈值以上的结果
- 不能挤掉默认结构检索的基础结果
- 命中的事件也优先按事件链片段方式注入

### 7.7 向量缓存

如果启用 AI 增强检索，需要为事件和档案建立本地向量缓存。

初期不引入外部向量数据库，优先使用 IndexedDB / Dexie 表保存：

- 目标类型：event / archive
- 目标 ID
- embedding model
- content hash
- vector
- updatedAt

事件或档案内容变更、embedding model 变更、content hash 变化时，再重新生成 embedding。

这样避免每轮都向量化整个事件池和档案库。

### 7.8 注入合并

最终注入内容由多路候选合并：

- 直接命中实体档案
- 当前在场实体档案
- 默认结构检索得到的事件链片段
- AI 增强检索得到的事件链片段
- 最终注入事件 `entityTags` 对应的实体档案
- 少量满足条件的桥接实体或语义命中档案

合并时应：

- 去重
- 结构检索优先
- AI 增强结果作为补充
- 数量上限可配置
- 初始阶段可以适当激进，后续根据实际噪声再下调

### 7.9 预设事件钩子

作者预设事件不属于运行时事实记忆层，而属于模组静态层中的剧情钩子。

当前推荐口径：

- `event catalog` 与运行时 `events` 明确分层
- 预设事件最小结构建议包含：`id / name / entityTags / content / trigger / guidance`
- `content` 是作者希望正文 AI 在合适时机吸收的剧情骨架
- `guidance` 是只给 AI 看的作者备注，用于说明吸收方向、偏差处理或分支倾向

触发方式当前推荐混合触发，两步处理：

1. 先用时间、`globals`、实体、地点等硬条件过滤掉明显不该出现的预设事件
2. 再结合当前实体池、当前在场实体、active event 和当前上下文，对剩余候选做软排序

注入方式当前固定为：

- 预设事件只注入给正文 AI
- 预设事件单独占一个分区，不与历史事件记录混在一起
- 预设事件进入上下文只表示“本轮可吸收”，不代表其已经成为事实

落成方式当前固定为：

- 如果正文没有实际采用该钩子，本轮不产生额外运行时状态变化
- 如果正文实际采用并形成事实，维护 AI 再按普通事实抽取逻辑创建或更新运行时事件
- 一旦落成运行时事件，后续就完全按 `active event -> 历史事件 -> 记忆检索` 主链处理

当前明确不做：

- 不给维护 AI 增加预设事件生命周期管理职责
- 不做预设事件运行时状态机
- 不增加 `sourceCatalogEventId`
- 不建立预设事件与运行时事件之间的程序级显式追踪关系

## 8. 关于“事件簇召回”与“精确锚定”

当前结论是：

- 默认结构检索优先找高质量 seed event，再扩展相邻事件链片段
- 不再把 AI 生成的实体查询词组作为主召回链路
- 时间仍主要作为排序、相邻事件扩展和消歧轴

因此：

- 事件召回不追求一次返回大量散点事件
- 更推荐返回少量可解释的连续事件链片段
- 如果没有高质量历史事件，允许不注入历史事件

这比单纯语义相似度排序更符合“维护剧情连续和逻辑通顺”的目标。

## 9. 注入策略

当前结论是：

`不做摘要压缩，只做数量控制。`

原因如下：

- 摘要会引入新的细节失真层
- 当前目标本来就是降低摘要带来的割裂感
- 现代模型对少量完整事件的理解能力足够强

因此，注入时：

- 直接注入完整事件或小段事件链
- 控制数量，但数量上限应可配置
- 初始阶段可以较激进，发现噪声后再下调
- 不额外压成短摘要

## 10. 当前不引入的概念

以下概念当前不作为正式系统概念引入：

- `强锚点`

原因如下：

- 它更适合作为分析术语，不适合作为正式系统抽象
- 引入后会增加抽象层级
- 对当前检索质量提升并不直接

当前更推荐使用的表达是：

- 当前实体池
- 默认结构检索
- seed event
- 相邻事件链片段
- AI 增强检索

## 11. 当前已确认的设计口径

截至目前，记忆系统的正式口径可以压缩为：

`事件记录为主存储，实体档案为辅助层，原文证据冷存储；事件维护支持轻量多 active，允许暂时没有 active event；事件标签当前只使用实际存在的叙事实体名称；name 是叙事标签而非唯一键；entityTags / linkedNames 是 AI 可读弱关系，内部 id 强关系只在当前可见实体池唯一解析时建立；检索默认走结构链路：由当前实体池匹配事件池，选择高质量 seed event，并沿时间线补相邻事件形成事件链片段；直接命中实体、当前在场实体、注入事件的 entityTags 档案默认注入；关联实体只作为弱信号或桥接候选；检索 AI 保留为可选 AI 增强检索，用非实体语义关键词和向量缓存召回补充事件/档案；不做摘要压缩，注入数量上限可配置。`

在事件记录表达上，进一步固定以下补充口径：

`事件记录同时保留结构标签与自然语言正文；结构标签负责检索、消歧和实体注入辅助；事件正文默认应包含足以锚定该事件的时间、地点、人物信息，并可直接作为注入内容使用。`

2. 档案层的表达边界

4. seed event 与相邻事件链片段的数量配置
5. AI 增强检索和向量缓存
6. 正式开发阶段的模块边界
