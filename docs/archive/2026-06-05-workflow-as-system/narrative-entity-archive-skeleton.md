# Narrative Entity Archive Skeleton

## 1. 文档目的

本文档用于确定`叙事实体档案`的骨架格式。

目标不是展开数据库设计，也不是定义所有世界观下的全部特殊字段，而是先确定：

- 哪些字段应当作为跨世界观通用字段固定存在
- 哪些字段属于实体变体扩展，而不是基础骨架
- 档案如何与事件和全局状态协作

## 2. 设计目标

叙事实体档案的目标不是承载完整剧情，而是作为`实体当前状态的唯一真源`。

因此档案骨架需要满足：

- 字段尽量少
- 语义表达自然
- 不依赖具体世界观
- 能同时适用于人物、地点、关键物件、组织等实体
- 能与事件记录稳定协作
- 能承接世界观专属扩展字段，但不把这些扩展写死进平台基础契约

## 3. 当前结论

当前建议将叙事实体档案收敛为：`扁平档案对象 + 极少系统字段 + 内部类型定义决定的扩展字段`。

进一步固定一个重要边界：

- 存储中的档案对象保持扁平，不按类型继承结构做嵌套存储
- 程序内部可以存在类似“父类 / 子类”的类型定义，用于复用字段定义
- AI 不需要看到内部继承关系，只看到最终实体类型及其允许维护的字段
- 前端包按具体玩法语义消费字段，平台不解释扩展字段内容

### 3.1 公共基础字段

这是对 AI 和人类都友好的主视图，字段尽量控制在最少数量。

固定字段如下：

1. `名称`
2. `别称与指代`
3. `背景`
4. `情境`
5. `关联实体`

补充口径如下：

- `关注点` 不再建议作为所有实体强制公共字段
- 角色、组织、怪物等可以把 `focus` 作为类型字段
- 地点、装备、材料、普通物品等可以没有 `focus`，或将其显示为更贴切的类型字段
- 存储层只保存字段本身，不要求所有实体拥有同一组业务字段

### 3.2 系统保留字段

系统内部只保留极少量必要字段，不让档案退化成元数据对象。

当前只建议预留：

1. `id`
2. `type`
3. `presence`

### 3.3 扩展字段

除公共基础字段和系统保留字段外，其他字段都视为`扩展字段`。

这些扩展字段具备以下口径：

- 扩展字段仍然属于档案本体
- 档案之外不再额外暴露一层“实体运行时状态”概念
- 档案对象本身就是实体当前状态
- 前端包直接消费档案对象渲染实体状态
- AI 注入时看到的实体信息，也应基于这一份档案视图

## 4. 为什么公共字段继续保持很少

### 4.1 名称

档案必须有稳定主称呼。

它是：

- 实体归一化的主锚点
- 事件标签引用时的主名称
- 检索与注入时的主要目标

### 4.2 别称与指代

这是当前档案骨架中最关键的辅助字段之一。

它用于承载：

- 别名
- 头衔
- 同义称呼
- 常见代称

它的作用不是丰富设定，而是提高玩家自然表达和系统命中之间的对接能力。

### 4.3 背景

`背景`承载低频变化、长期有效的信息。

它主要负责：

- 这个实体是什么
- 它在世界里处于什么位置
- 它过去是什么样

### 4.4 情境

`情境`承载当前剧情中可被认为成立的状态。

例如：

- 人物现在受伤、失踪、敌对、可信
- 地点现在封锁、失火、废弃、公开
- 物件现在归谁持有、是否损坏、是否暴露

### 4.5 关注点

`关注点`用于承载尚未收束、仍在剧情中继续起作用的内容。

它不是严格任务系统，也不是硬性待办，而是：

- 这个实体当前最值得继续追踪的事
- 它还挂着什么风险、目标、秘密、悬点

新的推荐口径如下：

- `关注点` 是部分实体类型的可选字段，不再是所有实体的基础字段
- 角色可以把它显示为“当前目标”
- 组织可以把它显示为“当前诉求”
- 地点、材料、装备等实体可以完全不提供该字段
- 这类差异由类型字段定义决定，而不是由平台基础档案结构强制规定

### 4.6 关联实体

`关联实体`用于承载这个实体当前最稳定、最强的叙事关联对象。

它当前不是关系图，也不是关系数据库，而是一组轻量的实体名称列表。

它的目标不是穷举所有关系，而是只保留那些足以影响检索与剧情连续性的强关联。

## 5. 类型定义与扁平存储

当前结论是：

`程序内部可以用类型继承复用字段定义，但存储对象仍然保持扁平。`

推荐理解方式如下：

- `EntityBase` 负责定义所有实体共有字段，例如名称、别称、背景、当前状态、档案条目状态、关联实体
- `Item` 可以在内部继承 `EntityBase`，并增加数量字段
- `Equipment` 可以在内部继承 `Item`，并增加品质、效果、属性加成等字段
- `Consumable` 可以在内部继承 `Item`，并增加使用效果、剩余次数等字段
- `Location`、`Character`、`Monster` 等也可以拥有自己的字段定义

但这些继承关系不直接暴露给 AI，也不直接进入存储结构。

存储中的档案仍然类似：

```ts
{
  id: "I8ZT",
  type: "equipment",
  name: "霜铃剑",
  aliases: ["旧剑"],
  background: "...",
  situation: "...",
  presence: "foreground",
  linkedNames: ["白沙"],
  quantity: 1,
  quality: "rare",
  effects: ["破阵"],
  statBonuses: { attack: 3 }
}
```

也就是说：

- 类型继承用于生成“这个类型最终有哪些字段”
- AI 只看最终扁平类型和合法字段列表
- 存储只保存扁平对象
- 平台不做复杂对象继承实例化，也不解释字段业务语义

## 6. 系统保留字段的进一步口径

### 6.1 id

`id` 只用于稳定定位一份档案，不承载语义正文。

当前结论如下：

- `id` 由程序生成，不由 AI 生成
- `id` 默认在单存档内唯一，不追求全局唯一
- 更新已有档案时，AI 应优先使用已注入给它的 `id`
- 创建新档案时，AI 不需要提供 `id`

当前推荐格式为：

`单字母基础类前缀 + 3 位随机字符`

推荐前缀如下：

- `C` 人物类
- `L` 地点类
- `I` 物件类
- `O` 组织类
- `X` 其他类

例如：

- `C7KQ`
- `L2MF`
- `I8ZT`

补充口径如下：

- 前缀只用于可读性，不作为完整类型真值来源
- 前缀只看 `type` 映射出的基础分组
- 若程序生成时发生冲突，则直接重生一个新 `id`

### 6.2 type

`type` 用于表示该实体当前属于哪种最终实体类型。

当前推荐口径为：

`扁平类型 id`

例如：

- `character`
- `monster`
- `location`
- `equipment`
- `consumable`
- `material`
- `organization`
- `clue`

进一步固定如下：

- AI 侧只看到这些最终类型，不看到父类 / 子类继承链
- 程序内部可以知道 `equipment` 继承自 `item`，`item` 继承自 `entityBase`
- 给 AI 注入或要求 AI 输出 patch 时，应展示合并后的最终字段列表
- `type` 承担“实体属于哪种最终类型”和“允许哪些扩展字段”的入口作用

平台层仍可保留少量基础分组用于生成 id 前缀、筛选或 UI 分组，例如：

- `character`
- `location`
- `item`
- `organization`
- `other`

### 6.3 presence

`presence` 用于表示该档案在记忆系统中的参与位置，而不是实体在剧情中的现实状况。

当前固定采用三态：

1. `foreground`
2. `background`
3. `retired`

## 7. 扩展字段的边界

当前结论是：

`扩展字段不是平台基础字段集合的一部分，但它们属于档案对象本体。`

这意味着：

- 平台不为所有实体预设一套统一的大字段全集
- 不同实体可以拥有不同扩展字段
- 同一基础类下的不同变体，也可以拥有不同扩展字段
- 即使同一变体下，不同实体也允许稀疏存在不同字段

进一步固定如下：

- 扩展字段的推荐范围由 `type` 对应的最终字段定义约束
- 当前原型期平台不需要实现复杂存储校验，只需保持对象可存储、可注入、可返回
- 平台不负责理解这些字段的具体语义
- 字段值可被前端包按自己的逻辑解释和渲染

可以压缩为一句话：

`平台统一字段写入契约，不统一字段内容语义。`

## 8. 档案、事件与全局状态的关系

当前建议如下：

- `事件`负责记录“发生过什么”
- `档案`负责记录“这个实体现在是什么样”
- `全局状态`负责记录“不属于单个实体但影响当前局面的状态”

也就是说：

- 事件是剧情主骨架
- 档案是实体当前状态
- 全局状态是会话侧补充

这里进一步固定一个重要口径：

`实体档案就是实体当前状态的唯一真源。`

因此：

- 对外契约中不再单独暴露“实体运行时状态”这一层
- 前端包渲染实体状态时，直接消费档案对象
- AI 注入实体信息时，也应基于档案视图

## 9. 档案更新的当前推荐口径

当前建议如下：

- 已有档案更新应优先按 `id` 定位
- 没有 `id` 的档案项默认视为新建候选
- 档案软删除不引入额外动作，直接通过 `presence = retired` 表达
- 暂时退出前台但仍活跃在背景中的档案，通过 `presence = background` 表达
- `关联实体` 由维护 AI 维护，不由程序硬编码推导
- 档案扩展字段的可写范围由 `type` 对应的最终字段定义约束

与 patch 契约相关的进一步口径，单独记录在：

- [patch-contract-skeleton.md](./patch-contract-skeleton.md)

## 10. 关于关键物件

当前结论是：

关键物件可以进入档案层，但不要求所有物件都进入。

一旦进入档案层：

- 仍然沿用同一套公共基础字段
- 可以再挂载 `type` 对应的扩展字段

这意味着：

- 人物和物件不需要两套完全不同的基础骨架
- 物件特有信息由扩展字段解决，而不是靠拆基础 schema 解决

## 11. 当前推荐口径

截至目前，叙事实体档案的推荐口径可以压缩为：

`所有实体档案在存储层保持扁平对象；平台基础字段只保留名称、别称与指代、背景、当前状态、关联实体，以及 id / type / presence 等极少系统字段；程序内部可以用父类 / 子类式类型定义复用字段，但 AI 只看到最终实体类型和合并后的合法字段列表；实体扩展字段不单独形成“实体运行时状态”层，而是直接作为档案对象的一部分存在；前端包直接消费档案对象渲染实体状态。`

## 12. 后续设计入口

在档案骨架已经收敛的前提下，后续可以继续围绕这些主题展开：

1. `type` 对应字段定义的最小契约
2. 档案更新 patch 的正式结构
3. 查询关键词组如何优先映射到档案实体
4. 关键物件进入档案层的提示词规则
5. 关联实体的维护口径与扩展边界
