# Tsian Next Implementation Plan

## 1. 文档目的

本文档记录从当前可运行原型继续推进的实施顺序。

旧版实施计划中的阶段 1 到阶段 9 已经基本完成或被当前实现覆盖，因此本文档不再重复早期骨架阶段。

当前目标是：

- 继续保持每一步都有可见效果
- 先验收已落地的新契约
- 再补平台 WebUI 的必要配置能力
- 避免在主链未稳定前提前扩展工坊、插件、复杂兼容层

## 2. 当前基线

当前项目基线如下：

- 平台 WebUI 可以启动
- 官方默认游玩前端包可以加载
- 本地 Dexie/IndexedDB 存档可用
- 已有检索 AI、正文 AI、维护 AI 最小链路
- 已有事件记忆、档案、globals、当前时间
- 已有默认测试种子
- 已有记忆检索调试面板
- 新 patch 契约已经在代码中第一轮落地
- 已有完整状态 checkpoint 最小骨架，支持列出和恢复线性状态切片
- 已有记忆检索 v2 第一轮落地：默认结构检索为基础链路，可选 AI 增强检索和本地向量缓存为补充链路

已验证命令：

- `npm run build:contracts`
- `npm run build:runtime-core`
- `npm run build:web`

详细当前状态见：

- `./current-state-handoff.md`

## 3. 当前实施原则

后续继续遵循：

- 先验收主链，再补配置 UI
- 先做最小可见入口，再扩完整编辑体验
- 先让用户能手动配置并跑通，再做预设、模组、工坊
- 平台只负责静态内容装载、运行时读写和 AI 调度，不统一承担玩法开局引导
- 如果某个玩法需要开局引导，由对应前端包自行实现，平台只提供其需要的桥 API
- 不做旧 IndexedDB 迁移
- 不做兼容层
- 不做复杂通用 patch 系统
- 不提前做多事件并行维护

## 4. 下一阶段实施顺序

### 阶段 A：真实运行验收新契约

目标：

- 确认当前代码和新文档契约真实一致
- 确认维护 AI 能稳定输出新 patch

验收标准：

- `npm run dev:web` 后页面能正常打开
- 新建存档后使用的是当前本地库
- 发送一轮真实游玩输入后，正文 AI 正常回复
- 控制台维护 AI 响应包含 `currentTime / globals / events / archives` 中的合理字段
- 事件更新走 `target: "active" + set`
- 档案更新走 `target + set` 或 `create`
- 快照中只有 `globals`，不再出现旧 `status`
- 档案中使用 `type / linkedNames`，不再出现旧 `kind / entityType / relatedEntities`

当前不要做：

- 不新增复杂 UI
- 不扩展 patch 动作
- 不补迁移
- 不重写检索策略

### 阶段 A2：档案 type 契约迁移

状态：已完成第一轮代码落地。

目标：

- 已将当前 `kind` 字符串迁移为最终实体 `type`
- 保持档案对象扁平存储
- 让内部类型定义可以表达父类 / 子类字段复用，但不暴露继承关系给 AI

验收标准：

- `ArchiveRecord` 使用 `type` 表示最终实体类型
- 公共字段不再强制包含 `focus`
- 类型字段定义能合并出给 AI 使用的最终字段列表
- AI patch 中 archives 仍然只写扁平字段，不出现继承结构或嵌套类型实例

当前不要做：

- 不做完整类型编辑器 UI
- 不做复杂存储校验器
- 不做旧 IndexedDB 迁移
- 不把内部继承树展示给 AI

### 阶段 B：AI 配置最小可视化入口

目标：

- 不再依赖手改本地配置完成基础测试

验收标准：

- 平台 WebUI 能配置正文 AI、检索 AI、维护 AI 的基础参数
- 配置能保存到浏览器本地
- 发送消息时实际读取该配置
- 配置错误能在页面或控制台明确暴露

当前不要做：

- 不做多供应商高级编排
- 不做复杂密钥管理
- 不做云端账号体系

### 阶段 C：提示词预设最小入口

目标：

- 让三类核心 AI 的提示词从代码常量逐步迁出

验收标准：

- 平台 WebUI 能查看当前检索 AI、正文 AI、维护 AI 的提示词片段
- 能编辑并保存最小提示词覆盖内容
- 新一轮对话能使用修改后的提示词
- 默认提示词仍可恢复

当前不要做：

- 不做 SillyTavern 完整导入
- 不做预设市场
- 不做复杂变量系统

### 阶段 D：运行时数据查看与调试增强

目标：

- 让记忆系统调试更直接，方便继续验证设计

验收标准：

- 平台或官方前端能清楚查看当前快照
- 能查看完整事件列表
- 能查看完整档案列表
- 能查看 globals
- 能查看最近一次维护 patch 原文

当前不要做：

- 不做编辑器型后台
- 不做版本回滚
- 不做审计系统

### 阶段 E：模组最小装载

目标：

- 让“模组作为内容产品”开始成立
- 让模组静态内容与玩家运行时存档正式分层

验收标准：

- 至少存在一个内置模组定义
- 新建存档时可以选择模组
- 模组静态层至少能提供 `manifest / frontend config / entity type definitions / archive catalog / event catalog / globals defaults`
- 模组中的 `event catalog` 与运行时 `events` 明确分层，前者是作者预设事件定义，后者是已发生/进行中的事实记录
- `event catalog` 至少有最小结构口径：`content / trigger / guidance`
- 预设事件候选不会因为“被注入给正文 AI”就直接落成运行时事件
- 平台能基于模组创建存档壳，但不统一承担玩法开局引导
- 灰盐镇测试内容在本阶段可暂时作为内置测试模组使用
- 模组内容与运行中存档区分清楚

当前不要做：

- 不做平台统一开局引导系统
- 不做上传下载
- 不做工坊
- 不做版本兼容
- 不做复杂依赖和冲突处理

### 阶段 F：前端包桥 API 收敛

目标：

- 固定游玩前端包与平台 WebUI 的最小通信边界
- 为前端包自行实现玩法初始化、玩法 UI 和运行时交互提供正式入口

验收标准：

- 前端包能读取模组静态内容
- 前端包能读取快照、历史、事件、档案、检索调试信息
- 前端包能通过正式写入口修改运行时状态，例如 `turn / currentTime / globals / history / events / archives`
- 前端包能发送玩家输入
- 正文 AI 能单独接收到通过触发筛选后的预设事件钩子分区
- 前端包如需实现开局引导，可只依赖桥 API 自行组装初始运行时状态
- 内部 AI 调度、上下文组装、patch 应用仍不暴露给前端包
- 官方默认前端包只依赖这套桥 API

当前不要做：

- 不做第三方包沙箱
- 不做权限系统
- 不做远程包加载

当前已完成的最小落地点：

- 平台桥已支持 `mod-static`
- 平台桥已支持 `write-runtime`
- `write-runtime` 当前采用 `完整切片替换 + 省略字段保持不变`
- `write-runtime` 错误返回已统一为 `error.code / error.message / error.details`
- 成功写入后会自动落一条 `manual` checkpoint

## 5. 推荐当前下一步

当前最建议先做：

`存档级 workflow preset override。`

原因：

- 当前 AIRP 主方向已经收敛为工作流编辑和运行平台
- workflow preset 已经能作为资源被编辑，也能通过 `ModManifest.workflowPresetId` 进入运行时
- 但玩家还不能为单个存档选择不同 workflow，这会把工作流能力限制在模组作者侧
- 存档级覆盖是最小、直接、可验证的一步，能让 workflow preset 正式成为玩家可选择的运行策略

本轮之后推荐顺序：

1. 存档级 workflow preset override
2. 节点输入/输出 schema 与语义槽选择
3. workflow run trace 和调试 UI
4. 将当前记忆链条逐步节点化，包括 retrieval、memory compose、maintenance、patch normalize、apply patch
5. 基础设施稳定后，再评估通用 MemoryStore 和第二类记忆架构

当前不要做：

- 不立即实现第二套记忆系统
- 不提前设计完整 MemoryStore 插件平台
- 不把平台存储、checkpoint、迁移、事务一致性和安全写回交给普通工作流节点

详细方向见：

- `./airp-workflow-platform-direction.md`
