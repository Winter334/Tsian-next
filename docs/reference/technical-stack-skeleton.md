# AIRP Technical Stack Skeleton

## 1. 文档目的

本文档用于确定项目的 `技术栈讨论骨架`。

当前阶段只回答以下问题：

- 技术栈需要围绕哪些核心点逐项讨论
- 每个核心点当前有哪些主要候选方向
- 哪些方向更适合当前的快速原型目标

当前阶段明确不做：

- 直接锁死最终技术选型
- 展开到具体库对比细节
- 展开到部署、监控、鉴权、CI/CD

## 2. 当前技术选型原则

当前项目的技术栈讨论，应固定在以下原则下进行：

- 以 `快速原型` 为优先目标
- 以 `记忆系统验证` 为优先目标
- 以 `单体系统优先` 为默认方向
- 优先选择简单、稳定、可持续迭代的组合
- 不为了未来可能的扩展提前拆成复杂服务
- 默认按 `本地运行时 + 官方平台分发` 路线思考

可以压缩为一句话：

`先用足够稳的技术把核心闭环做出来，再根据真实问题决定是否拆分。`

## 3. 当前待讨论的核心技术点

当前技术栈核心项已经全部收敛。

## 4. 后端选型结论

当前已确定：

- `Go`

原因很简单：

- 当前项目更像是高并发接口服务加 AI 调用编排
- 当前阶段更需要开发速度和实现效率
- Rust 更适合在后期证明某些模块确实需要更高性能时再考虑

这项结论当前可以压缩为：

`后端优先选择 Go，以服务快速原型、AI 调度编排与运行时接口开发。`

## 5. 前端选型结论

当前前端应分为两层：

### 5.1 平台 WebUI

当前已确定：

- `Vue 3`
- `TypeScript`
- `Vite`

原因如下：

- 平台 WebUI 负责配置、管理、发现、下载、上传与装载
- 它是稳定的官方壳层，而不是可被用户替换的游玩前端
- 这套组合足以承接固定管理界面与数据驱动平台逻辑

这项结论当前可以压缩为：

`平台 WebUI 选择 Vue 3 + TypeScript + Vite。`

### 5.2 游玩前端包

当前已确定：

- `不强绑技术栈`
- `只约束前端包产物与 API 契约`

原因如下：

- 官方游玩界面和玩家游玩界面在系统中地位一致
- 平台应统一装载前端包，而不是要求所有游玩前端必须同构
- 这样能最大化保持游玩前端的表现自由度

这项结论当前可以压缩为：

`游玩前端包技术栈无关，只认前端包产物与 API 契约；官方游玩界面只是平台内置的一种前端包。`

## 6. 官方平台数据库选型结论

当前已确定：

- `SQLite`

原因如下：

- 当前项目默认按 `本地运行时 + 官方平台分发` 路线推进
- 官方平台数据库主要承接平台、工坊、模组分发、账号与元信息
- 玩家核心游玩运行时默认不进入官方主库
- 当前项目不以扩展到大型中心化平台为目标

这项结论当前可以压缩为：

`官方平台数据库选择 SQLite，以服务轻量平台、工坊与分发能力。`

### 6.1 玩家本地运行时数据库

当前已确定：

- `IndexedDB`
- `Dexie`

原因如下：

- 玩家运行时默认在平台 WebUI 本地承载
- IndexedDB 更适合作为浏览器本地运行时数据库
- Dexie 更适合作为 IndexedDB 的工程化封装层

这项结论当前可以压缩为：

`玩家本地运行时数据库选择 IndexedDB + Dexie。`

## 7. 向量检索方案选型结论

向量检索的作用，是为记忆系统提供一层 `语义相近检索` 能力。

它不是关键词检索的替代，而是补充。

当前已确定：

- `本地轻量向量方案`

当前不作为第一阶段主方案的是：

- `独立向量数据库`
- `主数据库一体化向量方案`

原因如下：

- 当前项目的玩家运行时默认在本地
- 当前向量层主要服务于事件候选集的语义筛选，而不是全库主召回
- 当前阶段更需要验证记忆系统策略，而不是先建设本地向量数据库工程栈
- 本地轻量向量方案最符合快速原型目标

当前保留的升级方向是：

- 如果后期需要更强的一体化本地数据管理，可以升级到 `SQLite WASM + 向量扩展`
- 如果未来运行时重新回到本地服务进程或中心化服务端，再评估独立向量数据库

这项结论当前可以压缩为：

`第一阶段向量检索选择本地轻量向量方案；先验证事件候选集上的语义筛选能力，后续再视复杂度升级到 SQLite 一体化向量方案或独立向量库。`

## 8. 实时输出方向

当前这一点已经基本收敛为：

- `SSE`
- `非流式可切换`

当前讨论重点主要不是“要不要 SSE”，而是：

- 是否需要从一开始就支持流式和非流式两种模式
- 是否有必要引入 WebSocket

当前倾向很明确：

- `SSE + 非流式可切换`

## 9. 模组与存档的存储方式选型结论

当前已确定：

- `模组偏文件`
- `玩家存档偏本地`
- `模组分发走平台，玩家存档走本地`

原因如下：

- 模组本质上是内容产品，更适合作为可分发、可打包的独立内容单元
- 玩家存档本质上是本地运行时结果，默认不应由官方平台托管
- 这套方向最符合当前已经确定的 `本地运行时 + 官方平台分发` 路线
- 这样也更利于后续工坊分享、本地开发与可选云同步

这项结论当前可以压缩为：

`模组默认走文件化内容分发，玩家存档默认走本地存储；官方平台负责模组分发，不默认托管玩家存档。`

## 10. 当前不建议优先引入的内容

以下内容当前不建议优先进入第一阶段技术栈：

- 微服务拆分
- 独立工作流引擎
- 独立消息队列
- 独立缓存层
- 以 WebSocket 取代 SSE
- 一上来就独立向量数据库

原因如下：

- 会明显拉高系统复杂度
- 对当前核心验证目标帮助不大
- 很容易把精力从记忆系统验证转移出去
