# AIRP WebUI Runtime Skeleton

## 1. 文档目的

本文档用于确定项目的 `WebUI 运行层骨架`。

当前阶段只回答以下问题：

- WebUI 与后端的职责边界是什么
- 配置界面与游玩界面是否应分开
- 状态栏与游玩界面应如何接入后端
- “前端可自由改” 的边界应收敛到哪里

当前阶段明确不讨论：

- 具体前端技术栈
- 具体组件实现
- 页面视觉设计
- API 字段细节
- 权限、部署与鉴权

## 2. 当前结论

当前建议采用：

- `平台 WebUI`
- `游玩前端包`
- `前后端分离`
- `默认本地运行时`

也就是说：

- 系统相关配置通过一套平台 WebUI 完成
- 实际游玩时，平台 WebUI 本地持有核心运行时，并加载一个游玩前端包进行渲染
- 官方游玩界面和玩家游玩界面在系统里属于同一种前端包
- 前端包不直接介入 AI 调度、上下文组装与记忆处理细节
- 官方后端默认不承载玩家核心游玩运行时

可以压缩为一句话：

`平台 WebUI 持有本地运行时并负责管理与装载，游玩前端包负责实际渲染，官方后端主要负责平台与分发。`

## 3. 为什么这个方向成立

这个方向符合当前项目定位。

原因如下：

- 不同世界观的状态展示需求差异很大
- 状态栏不适合在底层被设计死
- 项目的核心竞争力不在固定 UI，而在记忆系统、AI 主链与预设体系
- 前后端分离更利于后续扩展不同风格的游玩界面
- 默认本地运行时可以显著降低官方服务器成本

## 4. 平台 WebUI 与游玩前端包的关系

当前建议明确分开两层前端：

### 4.1 平台 WebUI

平台 WebUI 是真正的官方前端。

它负责：

- 提供商配置
- AI 调度相关配置
- 预设管理
- 展示配置
- 其他系统级设置
- 模组发现、下载、上传与启用
- 游玩前端包发现、下载、上传与启用
- 玩家本地运行时管理
- 玩家本地数据持有

它本质上是：

- 平台管理界面
- 系统配置入口
- 内容装载入口
- 本地运行时壳层

它不是实际游玩的主要渲染界面。

### 4.2 游玩前端包

游玩前端包是实际承担游玩渲染的前端产物。

它通过平台 WebUI 提供的桥接能力获取：

- 对话内容
- 当前状态
- 状态栏所需数据
- 其他需要展示的运行时信息

它本质上是渲染层，不负责决定底层系统逻辑。

### 4.3 官方游玩前端与玩家游玩前端

当前建议固定以下关系：

- 官方游玩界面也是一个游玩前端包
- 玩家自定义游玩界面也是一个游玩前端包
- 两者在系统中的地位一致

这样可以保证：

- 官方方案不是特权实现
- 玩家方案不是附属插件
- 平台只需要提供统一装载与 API 能力

## 5. 平台运行时边界

当前建议将游玩阶段的平台壳层与游玩前端包边界收敛为：

- `一个核心运行时快照`
- `少量深度查询接口`
- `必要的消息交互接口`

### 5.1 核心运行时快照

游玩前端包在主界面中，优先读取一份面向当前会话的运行时快照。

这份快照应表达：

- 当前 AI 回复
- 当前对话主视图
- 当前时间
- 当前可展示状态
- 当前相关档案与事件信息
- 其他与“这一刻怎么渲染界面”直接相关的信息

它的定位不是：

- 后端内部数据原样倾倒
- 全量历史数据打包
- 全量档案库打包

这一层解决的是：

- 前端不需要自己拼当前视图
- 发送消息后可以直接用新快照刷新界面
- 官方 UI 与自定义 UI 都能依赖同一份稳定输入

### 5.2 深度查询接口

游玩前端包仍然可以读取平台运行时提供的更深层数据，例如：

- 完整档案数据
- 完整历史正文
- 完整事件记录
- 其他需要独立查看的完整信息

这一类接口的作用是：

- 支撑前端做档案页、时间线页、数据查看页
- 支撑更自由的可视化浏览能力
- 避免把主游玩快照做得无限膨胀

### 5.3 交互型接口

游玩前端包至少需要一个面向游玩的消息交互入口。

例如：

- 发送玩家输入
- 获取本轮生成后的新快照

这一类接口的作用是：

- 让前端可以驱动一轮游玩
- 让前端在一次交互后直接拿到新的运行时结果
- 不要求前端知道内部是如何完成这一轮处理的

### 5.4 不暴露给前端的内部能力

以下能力当前建议明确保留在平台 WebUI 的本地运行时内部：

- AI 调度
- 上下文组装
- 记忆检索主链
- 维护与沉淀流程
- Patch 应用过程

也就是说：

- 前端可以拿结果
- 前端可以发起交互
- 前端不直接操控内部运行链条

### 5.5 前端包与 API 契约

当前建议对游玩前端包只施加两类约束：

- 前端包产物约束
- 平台运行时桥接接口约束

这意味着：

- 平台不要求所有游玩前端包必须使用同一种技术栈
- 只要最终产物能被平台装载，并能按约定调用平台运行时桥接接口，即可接入框架能力
- 如果某个前端包完全不使用框架 API，它本质上就是独立前端，平台不为其内部逻辑负责

## 6. 状态栏的定位

当前建议不要把状态栏理解成固定字段集合。

更合理的定位是：

- 后端输出可供展示的状态信息
- 前端决定哪些显示、如何显示、以什么形式显示

这意味着：

- 核心状态不必全部显示
- 不同世界观可以有不同显示方式
- 同一套后端状态也可以被不同 UI 风格重组

## 7. “前端可自由改” 的边界

这部分必须收紧。

当前推荐边界是：

- 前端可以自由决定展示方式
- 前端可以自由决定布局与状态栏样式
- 前端不应承载核心运行逻辑

也就是说：

- 记忆系统
- AI 调度
- 时间推进
- 档案更新
- 事件沉淀

这些仍然应由平台 WebUI 的本地运行时负责。

否则系统会重新滑向：

- 前后端各自维护一套逻辑
- 系统边界混乱
- 后续维护成本快速上升

## 8. 实现难度的客观判断

如果目标是：

- 固定一套平台 WebUI
- 官方游玩界面作为一个标准前端包实现
- 状态栏通过快照数据驱动渲染
- 深度数据通过单独接口按需读取

那么实现难度是可控的，并不算夸张。

但如果目标变成：

- 让玩家自定义整套前端运行逻辑
- 让玩家上传前端脚本直接介入游玩主链

那么复杂度会明显上升，并且会很接近插件平台。

因此当前更推荐的方向是：

- 先做 `平台 WebUI + 前端包装载`
- 不做 `宿主内嵌插件式前端运行时`

## 9. 当前推荐口径

截至目前，WebUI 运行层的推荐口径可以压缩为：

`系统采用前后端分离；平台 WebUI 默认持有玩家本地运行时，并负责配置、管理、发现与装载；实际游玩由游玩前端包承担，官方游玩界面也是一种前端包；游玩前端以一份核心运行时快照作为主输入，并通过少量深度查询接口获取完整档案、事件与历史等数据；消息交互后直接返回新的运行时快照；AI 调度、上下文组装、记忆处理等内部主链默认保留在平台 WebUI 的本地运行时内部；游玩前端包默认只受前端包产物与 API 契约约束，不强绑技术栈。`

## 10. 后续设计入口

在 WebUI 运行层方向已经收敛后，后续最自然的问题是：

1. 平台本地运行时如何保存玩家数据
2. 平台如何装载和切换不同游玩前端包
3. 状态栏配置属于系统配置还是游玩主题配置
