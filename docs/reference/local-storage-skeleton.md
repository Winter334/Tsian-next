# AIRP Local Storage Skeleton

## 1. 文档目的

本文档用于确定项目中 `平台 WebUI 本地运行时数据存储` 的基础方向。

当前阶段只回答以下问题：

- 玩家本地运行时数据默认存在哪里
- 官方平台数据库与玩家本地数据库如何区分
- 哪种浏览器本地数据库方案更适合作为第一阶段主方案

当前阶段明确不讨论：

- 具体表结构
- 具体加密实现
- 多端同步细节
- 浏览器兼容边角问题

## 2. 当前结论

当前正式采用以下区分：

- `官方平台数据库 = SQLite`
- `玩家本地运行时数据库 = IndexedDB + Dexie`

可以压缩为一句话：

`平台库用 SQLite，玩家本地库用 IndexedDB + Dexie。`

## 3. 为什么需要单独区分

这两个数据库承担的职责完全不同。

### 3.1 官方平台数据库

官方平台数据库运行在官方后端。

它主要负责：

- 工坊
- 模组元信息
- 前端包元信息
- 平台配置
- 账号与平台级元信息

### 3.2 玩家本地运行时数据库

玩家本地运行时数据库运行在浏览器中，由平台 WebUI 持有。

它主要负责：

- 玩家存档
- 事件记录
- 档案
- 运行时快照
- 本地配置
- AI 提供商配置
- 本地敏感数据

## 4. 为什么第一阶段选择 IndexedDB

当前选择 IndexedDB 作为浏览器本地数据库的原因如下：

- 它是浏览器原生持久化数据库
- 更适合承载结构化本地数据
- 比 localStorage 更适合运行时数据与大量对象
- 不需要一开始就引入更重的本地数据库工程栈

## 5. 为什么选择 Dexie

当前建议将 Dexie 作为 IndexedDB 的主封装层。

原因如下：

- 它比原生 IndexedDB API 更适合工程开发
- 适合管理多类结构化数据
- 更适合作为平台 WebUI 本地运行时的主存储接口

因此第一阶段推荐口径是：

- `浏览器本地数据库选择 IndexedDB`
- `主封装层选择 Dexie`

## 6. 当前不推荐作为主方案的方向

以下方向当前不作为第一阶段主方案：

- `localStorage 作为主存储`
- `SQLite WASM 作为第一阶段本地数据库`

原因如下：

- localStorage 不适合作为本地运行时主库
- SQLite WASM 更适合作为后续本地数据工程化升级方向

## 7. 前端包与本地数据库的边界

当前建议固定以下关系：

- 平台 WebUI 自己管理 IndexedDB / Dexie
- 游玩前端包默认不直接拿到底层数据库实例
- 游玩前端包通过平台壳层桥接读写被允许的数据

这样可以避免：

- 游玩前端包直接接触敏感配置
- 游玩前端包直接操控底层本地存储
- 平台壳层与游玩前端包之间的边界被打穿

## 8. 当前推荐口径

截至目前，本地数据存储的推荐口径可以压缩为：

`玩家本地运行时数据库选择 IndexedDB + Dexie，由平台 WebUI 本地运行时统一管理；官方平台数据库仍然保持 SQLite；游玩前端包默认通过桥接能力访问被允许的数据，不直接持有底层数据库实例。`

## 9. 后续设计入口

在本地数据存储方向已经收敛后，后续最自然的问题是：

1. 平台本地数据库中的数据分层
2. 敏感数据与普通运行时数据的隔离口径
3. 未来何时从 IndexedDB + Dexie 升级到 SQLite WASM
