# Tsian Implementation Plan

## 1. 文档目的

本文档用于确定项目进入正式实现阶段后的 `实施顺序`。

目标不是罗列所有功能，而是：

- 保证每一步都有可见效果
- 保证实现顺序尽量低返工
- 避免过度设计与提前优化

当前阶段额外确认如下：

- 允许在需要时使用真实 API 进行测试
- 真实 API 配置可以先临时接入
- 后续再转为平台 WebUI 可视化配置

## 2. 实施原则

当前实施顺序固定遵循以下原则：

- 先跑通最小主线，再扩能力
- 先验证用户可感知效果，再做内部优化
- 先做稳定边界，再做不稳定策略
- 先做本地运行时闭环，再做平台增强能力

可以压缩为一句话：

`每一步都要能看见效果，但每一步只增加一类新复杂度。`

## 3. 当前已完成的骨架阶段

截至目前，以下骨架已经完成：

- 仓库目录骨架
- platform-server 最小 HTTP 服务
- platform-web 最小 Vue 壳
- contracts 最小契约骨架
- runtime-core 最小运行时入口
- platform-web 最小 runtime-host
- platform-web 最小 bridge
- platform-web 页面已能读取空快照

这意味着项目已经从“纯文档设计阶段”进入“可以开始跑最小实现链路”的阶段。

## 4. 推荐实施顺序

### 阶段 1：跑通现有骨架

目标：

- 让平台前端与后端真正启动成功

可见效果：

- `platform-web` 页面能打开
- 页面能显示当前空快照
- `platform-server` 的 `/` 和 `/healthz` 可访问

当前不要做：

- 不接 Dexie
- 不接 AI
- 不接前端包加载

### 阶段 2：接入本地存储最小壳

目标：

- 让平台 WebUI 真正持有本地运行时状态

可见效果：

- 页面能显示“本地数据库初始化成功”
- 刷新后简单本地值不丢
- 运行时快照可以落到本地数据库并重新读取

当前不要做：

- 不设计完整表结构
- 不接事件和档案复杂模型
- 不接 API Key 加密

### 阶段 3：跑通最小交互闭环，但先不用真实 AI

目标：

- 先让“发送消息 -> 新快照”这条主线成立

可见效果：

- 页面有输入框和发送按钮
- 发送后页面快照发生变化
- 假消息闭环能反复执行

当前不要做：

- 不接真实模型
- 不接检索
- 不接维护 AI

### 阶段 4：接入真实正文 AI

目标：

- 让项目第一次真正“能玩”

可见效果：

- 页面能保存本地 AI 配置
- 发送消息后返回真实 AI 文本
- 玩家能看到第一版真实剧情输出

当前不要做：

- 不接检索 AI
- 不接维护 AI
- 不做多提供商编排

### 阶段 5：接入官方默认前端包

目标：

- 从“平台页面里直接玩”切到“平台装载官方默认前端包来玩”

可见效果：

- 平台能加载官方默认游玩前端包
- 官方默认前端包能通过桥接读快照、发消息

当前不要做：

- 不做第三方前端包上传
- 不做复杂包隔离
- 不做版本兼容系统

### 阶段 6：接入最小本地存档

目标：

- 让“开一局”和“继续一局”真正成立

可见效果：

- 能新建存档
- 能进入不同存档
- 不同存档状态彼此独立

当前不要做：

- 不做云同步
- 不做分支/回滚
- 不做复杂导入导出

### 阶段 7：接入记忆系统第一版

目标：

- 先把“记住发生过什么”做出来

可见效果：

- 历史正文可以回看
- 关闭页面再打开还能继续
- 页面中能看到事件开始出现

当前不要做：

- 不做复杂事件筛选规则
- 不做完整档案更新体系
- 不做向量升级

### 阶段 8：接入档案第一版

目标：

- 让人物 / 地点 / 关键物件档案跑通

可见效果：

- 页面里能看到档案列表
- 某些角色或地点随剧情变化而更新

当前不要做：

- 不做复杂字段扩展
- 不做关系图
- 不做高级世界设定工具

### 阶段 9：接入本地轻量向量检索

目标：

- 让记忆系统开始具备更强语义筛选能力

可见效果：

- 候选事件排序结果更稳定
- 页面或调试面板能看到重排结果

当前不要做：

- 不做 SQLite WASM
- 不做独立向量库
- 不做全库 ANN

### 阶段 10：接入模组最小装载

目标：

- 让“模组作为内容产品”真正成立

可见效果：

- 平台可以装载内置模组
- 不同模组启动后内容不同

当前不要做：

- 不做工坊上传下载闭环
- 不做复杂依赖和冲突处理

## 5. 当前推荐执行顺序

当前推荐按以下顺序推进：

1. 跑通 `platform-web` 和 `platform-server`
2. 接 `IndexedDB + Dexie`
3. 做假消息闭环
4. 接真实正文 AI
5. 接官方默认前端包
6. 做本地存档
7. 做最小历史正文
8. 做最小事件沉淀
9. 做最小档案
10. 做本地轻量向量重排
11. 做模组装载
12. 最后再做工坊增强能力

## 6. 为什么这套顺序返工最少

原因如下：

- `平台壳` 比 `工坊系统` 更稳定
- `正文 AI` 比 `多 AI 联调` 更稳定
- `本地存档` 比 `云同步` 更稳定
- `事件主链` 比 `复杂档案扩展` 更稳定
- `候选集重排` 比 `完整向量工程化` 更稳定

换句话说：

- 先做确定性高、可见效果强的部分
- 再做复杂度高、策略性强的部分

## 7. 当前推荐口径

截至目前，实施计划的推荐口径可以压缩为：

`先跑通 平台壳 + 本地运行时 + 单轮交互 + 正文 AI 这条最小可玩主线，再逐层接入 Dexie、本地存档、事件、档案、轻量向量检索与模组装载；每一步都要求玩家可以直接看到效果。`
