# AIRP Local Runtime Skeleton

## 1. 文档目的

本文档用于确定项目的 `默认本地运行时` 定位。

当前阶段只回答以下问题：

- 玩家实际游玩时，核心运行时默认落在哪里
- 官方服务器默认承担什么，不承担什么
- 平台 WebUI 与游玩前端包在运行时中的关系是什么

当前阶段明确不讨论：

- 浏览器本地存储的具体实现
- 云同步细节
- 安全隔离细节
- 多端同步与冲突合并策略

## 2. 当前结论

当前正式采用以下方向：

- `玩家游玩运行时默认本地承载`
- `官方服务器默认不托管核心游玩运行时数据`
- `官方服务器主要提供平台与工坊能力`

可以压缩为一句话：

`平台负责分发与管理，玩家本地负责实际游玩运行时。`

## 3. 默认运行位置

当前建议将以下内容默认放在平台 WebUI 的本地运行时中：

- AI 调度
- 上下文组装
- 记忆检索
- 事件沉淀
- 档案更新
- 时间推进
- 玩家游玩存档
- 运行时快照

也就是说：

- 玩家发起一轮游玩时
- 默认不是把这一整套链路交给官方后端执行
- 而是由平台 WebUI 本地运行时完成

## 4. 官方服务器的默认职责

在当前方向下，官方服务器默认主要负责：

- 平台 WebUI 分发
- 模组分发
- 游玩前端包分发
- 工坊相关能力
- 可选的账号与同步能力

它默认不负责：

- 托管所有玩家的核心记忆库
- 托管所有玩家的核心档案库
- 托管所有玩家的每轮运行时状态
- 代替玩家承担主要 AI 调度成本

## 5. 平台 WebUI 与游玩前端包的关系

当前建议固定以下关系：

- 平台 WebUI 持有本地运行时
- 游玩前端包不直接拥有完整运行时
- 游玩前端包通过平台提供的桥接能力获取数据和发起交互

因此：

- 平台 WebUI 是运行时壳层
- 游玩前端包是表现层
- 官方游玩前端包与玩家游玩前端包共享同一类接入方式

## 6. 为什么采用这条路线

当前采用默认本地运行时，主要是因为：

- 官方服务器成本更可控
- 玩家 AI 提供商配置天然更适合本地持有
- AIRP 天然更接近单人本地运行时，而不是中心化多人在线游戏
- 工坊与游玩运行时可以彻底分离

## 7. 当前不排斥的增强方向

当前结论不是永久拒绝云能力。

而是：

- 第一阶段默认采用本地运行时
- 后续可以增加可选云同步或云托管能力

但这些内容不作为当前核心前提。

## 8. 当前推荐口径

截至目前，默认本地运行时的推荐口径可以压缩为：

`玩家实际游玩时的核心运行时默认由平台 WebUI 本地承载；官方服务器主要负责平台、工坊、模组与前端包分发；游玩前端包通过平台壳层获取运行时数据与交互能力。`

## 9. 后续设计入口

在默认本地运行时方向已经收敛后，后续最自然的问题是：

1. 平台 WebUI 本地运行时如何保存玩家数据
2. 游玩前端包与平台壳层之间的桥接边界
3. 官方平台数据库应只承载哪些数据
